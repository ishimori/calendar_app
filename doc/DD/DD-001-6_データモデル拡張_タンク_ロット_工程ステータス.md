## 目的

[DD-001-6/design/](DD-001-6/) で起こした高フィデリティ案（シート1=業務一覧 / シート2=ガント / 詳細編集モーダル）の実装に必要なデータ項目を、`JobData` JSON 構造の拡張として先行追加する。**画面のリデザイン本体は DD-001-7 に分離**し、本DDではモデル拡張とシード更新のみ行う。

## 背景・課題

- DD-001-5 までの `JobData` は Excel 運用の「行No・大区分・コメント・工程A〜E・Note・Event」を最小実装した形
- 高フィデリティ案では、生産管理者の実運用に必要な項目（タンクグループ／ロット名／担当／優先度／工程ラベル／工程ステータス／イベント種別）が追加で求められる
- これらは「画面の見た目」ではなくモデル側の拡張で、画面リデザインの前提条件になる

## 検討内容

### 1. デザイン案 vs 現行 JobData の差分マッピング

| 概念 | 現行 (`src/lib/schemas.ts`) | デザイン案 (`design/data.js` の `Row`) | 影響 |
|---|---|---|---|
| 行識別 | `rowNo: number` | `id: 'L-2401'` + `tank: 'T-01'` | `tank` / `lotName` を **新規追加**。`rowNo` は内部 PK として維持 |
| 大区分 | `category: string\|null` | `kubun` | 名前差異のみ。`category` を維持 |
| 担当 | なし | `owner: string` | **新規追加** |
| 優先度 | なし | `priority: '高'\|'中'\|'低'` | **新規追加** |
| 工程名 | `Step.name: 'A'..'E'` | `processes[].code: 'A'..'E'` + `label: '前処理'` 等 | `Step.label` を **新規追加**（`name` は工程コードとして維持） |
| 工程ステータス | なし（色のみ） | `status: 'done'\|'running'\|'planned'\|'overdue'\|'blocked'` | `Step.status` を **新規追加**。`Step.color` とは独立 |
| サブタスク | `Note { id, startDate, endDate, text, color? }`（期間） | `subTasks: [{ day, label }]`（単日） | **現行 `Note` を維持**（上位互換）。デザインの単日 chip は「期間1日=subTask」として描画する想定 |
| イベント | `JobEvent { id, startDate, endDate, text, color? }` | `events[]: { start, end, label, kind: 'milestone'\|'event' }` | `kind` を **新規追加**（デフォルト `event`） |

### 2. DBへの影響度

- **Prisma `Job` テーブル**: **変更なし**。`{ rowNo, data: String(JSON), updatedAt }` のまま。中身を JSON で持つ DD-001 方針が引き続き有効
- **`JobData` JSON 構造**: 拡張（後方互換は捨てる ＝ DD-001-5 と同方針、`db:reset && db:seed` で初期化）
- **マイグレーション**: Prisma migrate 不要

→ **「DBスキーマ」への影響は実質ゼロ**。Zod schema と seed の更新だけで吸収できる。

### 3. タンクの表現方法（重要な設計判断）

候補:
- **A案**: `JobData.tank: string|null` を追加し、UI 側で同じ tank の行をグルーピング（テーブル `rowSpan`、ガント `vA-tank-group`）
- B案: `Tank` テーブルを別 entity として切り出し、`Job` に `tankId` 外部キーを張る

採用: **A案**。

理由:
- Excel 運用の「1行 = 1ロット」「タンク変更は 1セル編集」操作感と整合
- `Job.data` の JSON 拡張だけで完結し Prisma 無改修
- タンクのマスタ管理（追加・改名・削除）は当面不要。DD-001-7 で必要になれば再検討

### 4. ステータスと色の関係

- `Step.status`: 意味（完了/進行中/計画/遅延/保留）
- `Step.color`: 表示色（DD-001-5 で導入したユーザー任意指定）
- → 両者独立。デザイン側のステータス色（`done: #22a06b` 等）はテーマトークンとして UI 側で定義し、`color` が未指定なら `status` から派生、指定があればそれを優先する設計を DD-001-7 で実装する

### 5. DD-001-7 への分割理由

- レビューゲートが2回必要（モデル拡張合意 → 高フィデリティ画面合意）→ guides.md 「粒度=レビューゲート1回」に従い分割
- 画面側は タンクフィルタ・モーダル・レジェンド・ロット範囲インジケータ・3種バー スタイル・weekend/today カラム など多数の要素があり、1DD 200行制限を超える見込み
- モデル拡張を先行マージしておけば、DD-001-7 の画面実装を並行着手しやすい

## 決定事項

- **Prisma スキーマ無改修**。`JobData` JSON 構造のみ拡張
- `JobData` に追加: `tank: string|null`, `lotName: string|null`, `owner: string|null`, `priority: '高'|'中'|'低'|null`
- `Step` に追加: `label: string|null`, `status: 'done'|'running'|'planned'|'overdue'|'blocked'|null`
- `JobEvent` に追加: `kind: 'milestone'|'event'`（デフォルト `'event'`）
- 後方互換なし。Phase 1 完了時に `db:reset && db:seed`
- 画面リデザインは **DD-001-7** に分離（本DD完了後に起票）
- 本DDでは既存画面が新スキーマで壊れない最小修正のみ行う（新項目は非表示でOK）

## スコープ外

- 画面リデザイン全般（タンクフィルタ、レジェンド、ツールバー新UI、モーダル、ロット範囲インジケータ等） → DD-001-7
- 月またぎ／複数月ビュー
- API 楽観ロック
- Tank のマスタ管理画面
